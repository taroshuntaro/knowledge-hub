# 社内ナレッジ共有サービス（knowledge-hub）設計書

- **日付**: 2026-07-04
- **ステータス**: 設計承認待ち
- **バージョン**: V1 スコープ

## 1. 概要

社内向けのブログ中心ナレッジ共有サービス。エンジニアは技術記事を、管理部・人事は社内トピックを投稿し、全社員が検索・閲覧・コメントできる場を提供する。

### 背景・動機

- **コスト削減**: 既存 SaaS（Notion、Confluence 等）の人数課金を回避する。
- **セキュリティ・社内ポリシー**: データを自社管理下に置く。
- **可搬性**: オンプレでも AWS でも同一構成で運用できることを要件とする。

### 前提

- 想定ユーザー数: 〜100 人（全ログインユーザーが閲覧者かつ書き手になりうる）
- 実運用を目的とした本番ツール
- 開発・保守: TypeScript 統一、少人数保守を前提

## 2. スコープ

### V1 に含む

- 記事の作成・編集・下書き（自動保存）・公開・論理削除（ゴミ箱・復元）
- リビジョン（保存時スナップショット、本人による閲覧・復元）
- 切替式エディタ（Markdown ソースモード ⇔ リッチモード）
- 画像アップロード（ドラッグ&ドロップ / ペースト）
- カテゴリ（2 階層・admin 管理、記事に 1 つ必須）
- タグ（フリーフォーム、誰でも作成可）
- ピックアップ（admin による記事のトップページ固定表示）
- 日本語全文検索（タイトル + 本文 + タグ、スニペット付き、カテゴリ・タグ・著者絞り込み）
- コメント（フラット + 1 階層返信、Markdown 対応）
- リアクション（絵文字プリセット、トグル式）
- アプリ内通知（コメント・返信・リアクション・メンション）
- @メンション（記事本文・コメント内）
- ブックマーク（あとで読む + 一覧ページ）
- 著者プロフィールページ（アイコン・自己紹介・執筆記事一覧）
- アカウント設定（プロフィール編集・パスワード変更）
- 認証: OIDC 汎用 + メール/パスワード（招待制・パスワードリセット付き）
- 権限: member / admin の 2 ロール
- 管理画面: ユーザー招待・無効化・ロール変更、カテゴリ管理

### V1 に含まない（将来拡張）

- Wiki（ストック型構造化ドキュメント）— 将来別途設計
- RAG 検索 — 布石のみ（§9 参照）
- Slack 連携・メール通知
- 部署・グループ単位の閲覧制限（ACL）
- 監査ログ
- 記事リビジョンの差分（diff）表示
- 画像以外のファイル添付
- 限定公開・下書きの共有

## 3. アーキテクチャ

```
┌─────────────── Docker Compose / ECS ───────────────┐
│                                                     │
│  ┌─────────────────────────────┐                    │
│  │  app コンテナ (Node.js 24)   │                    │
│  │                             │    ┌────────────┐  │
│  │  Hono                       │───▶│ PostgreSQL │  │
│  │  ├─ /api/*   REST API       │    │ ・pg_bigm  │  │
│  │  ├─ /auth/*  認証           │    │ ・pgvector │  │
│  │  └─ /*       SPA 静的配信    │    └────────────┘  │
│  └─────────────────────────────┘                    │
│         │              │                            │
│         ▼ S3 API       ▼ SMTP                       │
│  MinIO / S3     社内SMTP / SES（開発: Mailpit）      │
└─────────────────────────────────────────────────────┘
```

### 技術スタック

| レイヤ | 技術 | 備考 |
|---|---|---|
| バックエンド | Hono (Node.js 24) | REST API + SPA 静的配信。RPC モードで型をフロントと共有 |
| フロントエンド | Vite + React SPA | 社内ツールのため SSR 不要 |
| エディタ | Tiptap + CodeMirror | リッチモード / Markdown ソースモード |
| DB | PostgreSQL 16+ | 拡張: pg_bigm（全文検索）、pgvector（将来 RAG） |
| ORM | Drizzle ORM / Drizzle Kit | マイグレーション管理込み |
| オブジェクトストレージ | S3 互換 API | オンプレ: MinIO / AWS: S3 |
| メール | SMTP（環境変数設定） | 招待・パスワードリセット用。開発は Mailpit |
| テスト | Vitest / Testcontainers / Playwright | §11 参照 |
| リポジトリ | pnpm workspace モノレポ | `apps/server` / `apps/web` / `packages/shared` |

### 可搬性（オンプレ / AWS）

- **オンプレ**: `docker-compose up`（app + PostgreSQL + MinIO + Mailpit or 社内 SMTP）
- **AWS**: 同一 app イメージを ECS / App Runner に配置。RDS for PostgreSQL（pg_bigm / pgvector 対応済み）+ S3 + SES
- 環境差分は**環境変数のみ**とする。コードに環境分岐を持ち込まない。

### モノレポ構成

- `apps/server`: Hono API。ルーティング → サービス層 → Drizzle の 3 層。
- `apps/web`: Vite + React SPA。
- `packages/shared`: Zod スキーマ（API 入出力の型・バリデーション）を両者で共有。

### 設計原則

- 記事の作成・更新・削除は必ずサービス層の `ArticleService` を通す（将来の RAG 同期・通知などのフック点を一箇所に集約）。
- 検索は `SearchService` インターフェースの背後に置く。V1 実装は pg_bigm のみ。
- 権限チェックは `can(user, action, resource)` 形式で一箇所に集約。
- 通知生成はサービス層 1 箇所に集約（将来の Slack 等の配信先追加に備える）。

## 4. データモデル

| テーブル | 主なカラム・役割 |
|---|---|
| `users` | email, display_name, avatar_url, bio, role (member/admin), auth_provider (oidc/password), password_hash (nullable), is_active |
| `sessions` | セッション ID（httpOnly Cookie 用）、user_id, expires_at |
| `invitations` | 招待トークン。email, token_hash, expires_at, used_at |
| `password_reset_tokens` | user_id, token_hash, expires_at, used_at |
| `articles` | author_id, category_id（FK、nullable — 公開時に必須）, title, body_md（Markdown 原文・正）, search_text（平文化、pg_bigm GIN インデックス）, status (draft/published), published_at, updated_at, deleted_at（論理削除）, pinned_at（nullable、ピックアップ表示） |
| `article_revisions` | article_id, title, body_md, saved_at。保存時スナップショット |
| `categories` | name, parent_id（nullable、深さ最大 2）, sort_order。admin 管理 |
| `tags` / `article_tags` | タグ名（ユニーク）/ 多対多 |
| `comments` | article_id, author_id, parent_id (nullable, 1 階層), body_md, deleted_at |
| `reactions` | (user_id, article_id, emoji) ユニーク |
| `bookmarks` | (user_id, article_id) ユニーク |
| `notifications` | recipient_id, type (comment/reply/reaction/mention), actor_id, article_id, comment_id (nullable), read_at |
| `uploads` | uploader_id, s3_key, mime_type, size |

- **記事本文は Markdown 文字列（`body_md`）を正とする。** エディタ・検索・将来の RAG すべてがこれを源泉とする。
- 記事 URL は ID ベース（`/articles/:id`）。日本語タイトルのスラッグ化は行わない。
- 将来 RAG 用の `article_chunks`（チャンク + embedding）は V1 では**作成しない**。

## 5. 認証・権限

### 認証

- **OIDC 汎用**: issuer URL / client ID / secret を環境変数で設定し、任意の IdP（Google、Entra ID、Keycloak 等）に接続。初回ログイン時にユーザーを自動作成（JIT プロビジョニング、ロールは `member`）。
- **パスワード認証**: admin がメールアドレス指定で招待 → 招待リンクからパスワード設定。パスワードリセットはメール経由。環境変数でパスワード認証自体を無効化可能（OIDC 専用運用）。
- セッション: DB 保存のセッション ID を httpOnly / SameSite=Lax Cookie で保持。
- ログイン試行にレートリミットを適用。

### 権限（ロールベース 2 段階）

| ロール | できること |
|---|---|
| `member` | 記事の作成・公開、自分の記事・コメントの編集・削除、コメント・リアクション・ブックマーク |
| `admin` | member の全権限 + ユーザー管理（招待・無効化・ロール変更）、任意の記事・コメントの削除、カテゴリ管理、ピックアップ管理 |

- 下書きは本人と admin のみ閲覧可。公開記事は全ログインユーザーが閲覧可。
- **最後の admin は降格・無効化できない**（締め出し防止ガード）。
- ユーザー無効化はログイン不可にするのみ。**コンテンツと著者表示は残す**（退職者の知見を保全）。
- **ユーザー無効化・パスワード変更時は該当ユーザーの既存セッションを即時失効**させる。
- パスワード最低要件: 12 文字以上（文字種強制はしない。NIST SP 800-63B に従い長さを優先）。
- admin への昇格は管理画面から行う（OIDC のクレームマッピングは V1 では行わない）。
- 部署・グループ ACL は意図的に持たない。必要になった時点（Wiki 展開時）に `can()` の拡張として設計する。

## 6. エディタ

- 基盤は **Tiptap**。タブで「リッチモード（WYSIWYG）」と「Markdown ソースモード（CodeMirror + プレビュー）」を切替。
- **保存形式は常に Markdown。** リッチモードは Markdown をパースして編集し、Markdown にシリアライズして戻すビューとして実装する。
- 往復（Markdown → リッチ → Markdown）の無損失性を最優先とし、対応記法を以下に限定する。リッチモード側もこの範囲の機能のみ提供する:
  - 見出し（h1〜h3）、段落、太字・斜体・打消し・インラインコード
  - リンク、画像
  - 箇条書き・番号付きリスト・タスクリスト
  - 引用、水平線
  - コードブロック（言語指定 + シンタックスハイライト）
  - 表（GFM テーブル）
- 生 HTML は入力・保存とも許可しない。
- 画像はドラッグ&ドロップ / ペーストでアップロード → Markdown に URL 挿入。
- 下書きは自動保存（数秒デバウンス）。保存時に `article_revisions` にスナップショットを記録。

## 7. 情報設計（カテゴリ・画面構成）

### カテゴリ（2 階層）

チャネル型（投稿先を選ぶ・チャネルごとにフィードが分かれる）は採用しない。全員が同じ場に書き、**全体フィードを主役**に保つため、分類はカテゴリ（ラベル）方式とする。

- 深さ最大 2 の階層。例: テック ＞ フロントエンド / インフラ、人事 ＞ 制度 / 採用、お知らせ（子なしも可）。
- 記事には**カテゴリ 1 つ必須**（大分類・小分類どちらを付けてもよい）。必須チェックは**公開時**に行い、下書きの間は未設定を許す（自動保存を妨げないため）。
- タグはカテゴリ横断の補助分類（例: 「AWS」「新人向け」）としてフリーフォームのまま。
- カテゴリの作成・改名・並び替え・削除は admin のみ（散らかり防止）。
- **配下（小分類・記事）が空でないカテゴリは削除不可。** 削除時は移行先カテゴリを指定して記事を付け替える。
- カテゴリページは配下の小分類の記事も含めて表示する（「テック」を開けばフロントエンドの記事も出る）。
- 3 階層以上が必要になるのはストック型（Wiki）の兆候とみなし、V1 では拡張しない。

### 画面一覧

| 画面 | パス | 内容 |
|---|---|---|
| トップ | `/` | ピックアップセクション + 全記事の新着フィード（部門問わず時系列）。サイドバーにカテゴリツリー・人気タグ |
| カテゴリ | `/categories/:id` | 該当カテゴリ + 配下の記事一覧（時系列） |
| タグ | `/tags/:name` | タグ別記事一覧 |
| 検索結果 | `/search` | 全文検索 + カテゴリ・タグ・著者絞り込み |
| 記事詳細 | `/articles/:id` | 本文、コメント、リアクション、ブックマーク |
| エディタ | `/articles/new`, `/articles/:id/edit` | 切替式エディタ、カテゴリ・タグ指定 |
| マイ記事 | `/me/articles` | 自分の下書き / 公開済み / ゴミ箱のタブ切替 |
| ブックマーク | `/me/bookmarks` | 自分のブックマーク一覧 |
| アカウント設定 | `/settings` | プロフィール編集（表示名・アイコン・bio）、パスワード変更 |
| プロフィール | `/users/:id` | アイコン・自己紹介・執筆記事一覧 |
| 通知 | `/notifications` | 通知一覧（ヘッダーのベルからも直近分を表示） |
| 管理画面 | `/admin` | ユーザー管理、カテゴリ管理 |
| 認証系 | `/login` ほか | ログイン、招待受諾、パスワードリセット |

### UI 方針

- 一覧はカーソルベースのページネーション（無限スクロールではなく「もっと見る」）。
- 閲覧系画面はレスポンシブ対応（スマホで読める）。エディタはデスクトップ優先。
- 削除済み記事は本人と admin のみ直接 URL でアクセス可（削除済みバナー表示）。復元・完全削除はそこから行う。

## 8. 機能設計

### 記事ライフサイクル

- draft（本人 + admin のみ）→ published（全員閲覧可）。公開後も編集可（更新時もリビジョン記録）。
- 削除はゴミ箱（`deleted_at`）へ。本人と admin が復元可能。ゴミ箱からの物理削除は admin のみ。
- 本人は過去リビジョンの一覧・閲覧・復元が可能（diff 表示は V2）。

### ピックアップ（トップ固定表示）

- admin が記事詳細ページのメニューからピン留め・解除する。対象は公開記事のみ。
- トップページの新着フィード上部に「ピックアップ」セクションとして `pinned_at` 降順で表示。通常フィードからは除外しない（フィードは常に全記事の時系列、という §7 の原則を保つ）。
- **非公開化・ゴミ箱行きの際はピンを自動解除。** 解除は一方通行で、復元・再公開時に自動復活はしない（古いお知らせが意図せずトップに再登場する事故の防止）。
- 件数の上限は設けないが、運用ガイドとして常時 3〜5 件程度を README に記す。
- 期限付きピン（自動解除）・手動並び替えは V1 では作らない。

- `pg_bigm` GIN インデックスによる日本語部分一致検索。対象は `search_text`（タイトル + 本文平文 + タグ）。
- 結果はヒット箇所前後のスニペット付き。カテゴリ・タグ・著者での絞り込みと併用可。
- `SearchService` インターフェース背後に実装（将来の RAG 差し込み点）。

### コメント・リアクション・ブックマーク

- コメント: フラット + 1 階層の返信。Markdown 対応（簡易テキストエリア、エディタ切替なし）。削除は論理削除（「削除されました」表示で返信ツリーを保つ）。
- リアクション: 絵文字プリセット数種類。ユーザー × 記事 × 絵文字でトグル。楽観的更新（失敗時ロールバック）。
- ブックマーク: トグル式。自分のブックマーク一覧ページを持つ。

### 通知・メンション

- アプリ内通知（ベルアイコン + 未読バッジ、既読管理）。
- トリガー: 自分の記事へのコメント / リアクション、自分のコメントへの返信、@メンション（記事本文・コメント内）。
- 新着記事は通知しない（トップページの時系列フィードが担う）。
- 自分自身の行動では通知を生成しない。

### プロフィール

- アイコン（アップロード）、表示名、自己紹介（bio）、執筆記事一覧。

### 画像アップロード・配信

- 画像のみ許可（MIME タイプ検証・サイズ上限）。
- Markdown にはアプリ経由の安定 URL（`/api/uploads/:id`）を埋め込む。エンドポイントが認証チェック後に S3 から配信（ストリーミングまたは短期署名付き URL へのリダイレクト）。署名付き URL を直接 Markdown に埋め込むことはしない（期限切れで画像が壊れるため）。

## 9. 将来の RAG 対応（布石のみ）

RAG の実現方式は未決定（スタック内 pgvector or Bedrock ナレッジベース + S3 Vectors）。V1 は**どちらも塞がない**状態を保つ:

- ソースコンテンツ: `articles.body_md`（Markdown 正）が源泉。
- 増分同期: 安定した記事 ID + `updated_at` + リビジョンで対応可能。
- フック点: 記事の作成・更新は `ArticleService` を必ず通るため、そこがチャンク化（pgvector 方式）または S3 エクスポート（Bedrock 方式）のトリガー挿入点になる。
- 検索の差し込み: `SearchService` インターフェースに実装を追加する形で対応。
- 注意: Bedrock 方式は AWS 依存のため、オンプレ本番の場合は pgvector 方式が安全側の選択となる（将来判断）。

## 10. エラー処理

- API は統一エラー形式 `{ code, message, details? }`。バリデーション（Zod）400 / 未認証 401 / 権限なし 403 / 不在 404 / サーバー内部 500。
- フロント: フォームエラーはフィールド横、それ以外はトースト表示。
- 楽観的更新はリアクションのみ。記事保存はサーバー確定を待つ（自動保存のため体感影響なし）。
- 編集競合: 他人の記事は編集不可のため、`updated_at` 比較の楽観ロックで「別の場所で更新されています」と警告する最小対応のみ（複数タブ対策）。

## 11. セキュリティ

- Markdown レンダリングはサニタイズ必須（rehype-sanitize）。生 HTML 不許可。
- CSRF: SameSite=Lax Cookie + Origin ヘッダ検証。
- アップロード: MIME タイプ検証・サイズ上限・画像のみ。配信は認証必須（§8）。
- ログイン試行のレートリミット。
- セキュリティヘッダ（CSP 等）。
- 依存パッケージの脆弱性チェックを CI に組み込む。

## 12. テスト戦略

- **ユニット / 統合**: Vitest。サービス層（`can()` 権限チェック含む）を重点対象。API 統合テストは Testcontainers で実 PostgreSQL を起動し、pg_bigm の検索クエリまで実物で検証。
- **エディタ往復変換テスト**: 「Markdown → リッチ → Markdown」が同一になることをサンプル群（§6 の全対応記法を網羅）で検証する専用スイート。切替式エディタの品質保証の要。
- **E2E**: Playwright。主要フロー（ログイン → 記事作成 → 公開 → 検索 → コメント → 通知）のみ薄く。

## 13. 運用

- ヘルスチェック: `/healthz`（DB 接続確認込み）。
- ログ: 構造化ログ（pino、JSON）。オンプレは docker logs、AWS は CloudWatch へそのまま。
- マイグレーション: デプロイ時に `drizzle-kit migrate` を明示実行。
- バックアップ: オンプレは pg_dump 日次 cron + MinIO データ、AWS は RDS スナップショット + S3 バージョニング。手順を README に明記。
