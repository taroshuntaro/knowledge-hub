# EditorPage 刷新（記事エディタの執筆没入化）設計

- 日付: 2026-07-07
- ステータス: 承認済み（実装前）
- 位置づけ: UI/UX 刷新の追加サブフェーズ **S5**（S1〜S4 完了後）。記事を生み出す中心導線であるエディタを、執筆に没入できるレイアウトへ刷新する。

## 1. 背景・目的

S1〜S4 でトークン（インディゴ）・アプリシェル・統一カード・記事詳細/一貫性は刷新済みだが、`EditorPage` は S1 のシェル・トークンの恩恵を受けただけで、執筆体験としてのレイアウト・情報設計は未刷新のまま残っている。実アプリのスクリーンショットでも裏取り済みの主な課題:

1. 保存状態が不明瞭（2 秒デバウンス自動保存が走るのに常時インジケータがない）
2. アクション（下書き保存/公開）が本文末尾に埋没し、長文だと届きにくい
3. 公開/下書きの現在状態が画面から判別できない
4. 離脱導線（戻る）がない
5. タイトル→ヒーロー→カテゴリ→タグの縦積みフォーム 4 段を越えないと本文に届かず、**「先に埋めなければ」という心理的障壁が執筆を妨げる**。特に未設定でも巨大な 16:9 ヒーロー空枠が上部を占有している
6. タイトルが通常 Input で、執筆画面としての没入感が薄い

目的は、**「完成した記事詳細ページをその場で編集している」ようなキャンバス**にし、分類用の管理メタ（カテゴリ/タグ）を執筆の前提から外して公開時に集約すること。既存の堅牢な保存・変換ロジックは無改変で維持する。

## 2. スコープ

### 対象
- `apps/web/src/pages/EditorPage.tsx`（レイアウト・状態表示・公開パネル）
- `apps/web/src/components/HeroImageInput.tsx`（16:9 contain＋ぼかし背景、未設定コンパクト化）
- `apps/web/src/pages/ArticleDetailPage.tsx`（ヒーロー表示を contain＋ぼかし背景に統一）
- `apps/web/src/components/ArticleCard.tsx`（サムネ枠を 4:3 cover へ）
- 上記に付随するテスト

### 非対象
- サーバー API・DB スキーマの変更（`heroImageUploadId`・`status`・`publishedAt`・publish 時カテゴリ必須は既存のものをそのまま使う）
- リッチ/Markdown 変換ロジック、自動保存の直列化、画像 D&D/ペーストの挙動変更
- 記事ごとのアスペクト比選択（heroAspect 列などは**作らない**）

## 3. 表示比率の方針（確定事項）

画像ファイル自体は任意比率で受け付ける。**表示枠の比率を用途ごとに固定**し、`object-fit` で吸収する。

| 用途 | 表示枠 | fit | 挙動 |
|---|---|---|---|
| ヒーロー（記事詳細・エディタ入力プレビュー） | **16:9 固定** | **contain** | 16:9 画像は余白なし。4:3 など縦長は高さを 16:9 に合わせ**左右余白**。余白は**同画像のぼかし背景（blur backdrop）**で自然に埋める。画像は切れない。 |
| サムネイル（一覧カード左サムネ） | **4:3 固定** | **cover** | 4:3 画像はぴったり。16:9 画像は高さ合わせで**左右がトリミング**（許容）。一覧の並びを優先し充填する。 |

ヒーローを contain にする狙いは「高さを 16:9 基準に抑え、どの記事もヒーロー高さが揃い上部がすっきりする」こと。サムネを cover にする狙いは「小さな認識用の絵は余白より充填が見栄えする・並びが揃う」こと。

## 4. レイアウト構造

アプリシェル（サイドバー/ドロワー）の内側で、`EditorPage` を次の 2 層にする。

```
┌ sticky アクションバー（上部固定） ───────────────────────────────┐
│ ← 戻る │ [ステータスバッジ] │ [自動保存インジケータ]  …  [下書き保存][公開する ▸] │
├────────────────────────────────────────────────────────┤
│ キャンバス（縦スクロール, max-width ~720px, 中央寄せ）              │
│   [ヒーロー: 未設定=1行トリガー / 設定=16:9 contain＋ぼかし背景]     │
│   # タイトル（見出し風入力）                                      │
│   著者アバター・下書き状態などのメタ行（プレビュー）                 │
│   ── 区切り ──                                                  │
│   [リッチ | Markdown] セグメント                                  │
│   本文（記事ページと同じ幅・タイポ）                               │
└────────────────────────────────────────────────────────┘
```

## 5. sticky アクションバー

- 位置: キャンバス上部に `position: sticky; top: 0`。スクロールしても常に届く。半透明背景＋blur。下境界線。
- 左側:
  - **「← 戻る」**: 既存記事編集なら `/articles/:id`、新規なら `/`。未保存があれば保存フラッシュを試みてから遷移（実装者判断で `history.back()` ではなく明示ルートに寄せる）。
  - **ステータスバッジ**: `status === 'draft'` なら「下書き」（secondary）、`'published'` なら「公開済み」（緑系・既存トークンで表現）。新規未保存は「下書き」。
  - **自動保存インジケータ**: 状態に応じて「保存中…」/「◯分前に保存」/「未保存」/「保存に失敗」を表示。既存の `status`/`error` state と保存タイミングから導出（§9）。
- 右側:
  - **「下書き保存」**（outline）: 既存 `enqueueSave` を呼ぶ。いつでも押せる。
  - **「公開する ▸」**（primary）: 公開パネルを開く（§7）。公開済み記事の編集時は主ボタンのラベルを **「更新を公開」** にする。

## 6. ヒーロー画像（キャンバス上部）

- **未設定時**: 上部を占有しない**コンパクトな 1 行トリガー**「🖼 ヒーロー画像を追加（任意）」。クリックでファイル選択。書き始めに事前設定を強いない。
- **設定時**: **16:9 枠・contain・ぼかし背景**で表示（§3）。ホバー/フォーカスで「画像を変更」「削除」操作を出す。
- `HeroImageInput` をこの仕様に作り替える（現状は 16:9 の破線ボックス＋object-cover＋常時巨大枠）。記事詳細の見え方と一致させるため、ぼかし背景付き contain のプレビューを共通の見た目にする。
- **記事詳細ページ**（`ArticleDetailPage`）のヒーローも同じ 16:9 contain＋ぼかし背景に変更し、エディタのプレビューと実際の表示を一致させる。

### ぼかし背景の実装指針
16:9 枠を `relative overflow-hidden`。枠内に 2 レイヤー:
1. 背景: 同じ画像を `absolute inset-0 object-cover` で敷き、`filter: blur(...)` ＋軽い暗さ／拡大（`scale`）でにじませる。`aria-hidden`。
2. 前景: 実画像を `object-contain` で中央配置（`h-full` 基準）。`alt` は記事タイトル等。

16:9 画像なら前景が枠を埋め背景は見えない。縦長画像なら左右にぼかし背景が覗く。エディタ入力・記事詳細で見た目を一致させるため、この表示は共通コンポーネントに切り出してよい（実装者判断で `HeroImage` 表示用コンポーネントを作り DRY 化）。

## 7. 公開パネル（カテゴリ/タグの集約）

- 執筆キャンバスから**カテゴリ・タグの入力欄を撤去**する。
- 「公開する ▸」/「更新を公開」押下で**公開パネル**を開く（右スライドのシート、またはダイアログ。既存の radix Dialog を用いてよい）。パネル内容:
  - **カテゴリ**（`CategorySelect`）: 見出しに「必須」表示。
  - **タグ**（`TagInput`）: 任意。
  - **公開/更新ボタン**: カテゴリ未選択なら**無効化＋⚠「公開にはカテゴリの選択が必要です」**。有効時に押すと、まず保存をフラッシュ（既存 `enqueueSave`）→ `publish` を実行 → 記事詳細へ遷移。
- カテゴリ必須は**サーバー `publishArticle` で既に担保**（`categoryId` 無しは 400）。クライアントの無効化は先回り UX であり、二重の守り。パネルを開いた時点でカテゴリ未設定なら警告を初期表示する。
- 下書き保存はパネルを介さずいつでも可能（カテゴリ未設定でも下書きは保存できる既存挙動を維持）。

## 8. サムネイル（一覧カード）

- `ArticleCard` の左サムネ枠を **4:3 固定（`aspect-[4/3]`）＋ `object-cover`** にする。
- 画像なしフォールバックタイル（カテゴリ色 or muted）の比率も 4:3 に合わせる（現状 `aspect-[16/9]`）。
- カード全体のレイアウト（左サムネ＋右本文、`items-stretch`）は維持しつつ、サムネの幅/高さがカードと破綻しないよう整える（サムネ基準で高さを決めるか、幅固定＋4:3 高さにするかは既存カードの伸縮と整合する形で実装者が調整）。

## 9. 保存・公開状態の可視化

現状 state（`status`, `error`, `updatedAt`）に加え、自動保存の進行を表す軽い派生状態を用意する（例: `'idle' | 'saving' | 'saved' | 'error'` と最終保存時刻）。デバウンス発火〜`save` の解決で遷移させ、アクションバーのインジケータに反映する。「◯分前に保存」は保存時刻からの相対表示（保存直後は「保存しました」→以降相対でよく、厳密な実時間更新までは不要）。既存の `enqueueSave`/直列化チェーン/2 秒デバウンスは**変更しない**（表示の被せのみ）。

## 10. 既存機能の維持（無改変）

- 自動保存の直列化（`enqueueSave`/`saveChain`/`saveRef`）と 2 秒デバウンス
- リッチ/Markdown 切替と `canEnterRich` の損失ガード（変換して続行/キャンセル）
- Markdown モードの左エディタ（CodeMirror）＋右プレビュー 2 カラム
- 画像 D&D/ペーストのアップロード（`handleSourceImageUpload`, `firstImageFile`）
- publish 時に保存をフラッシュしてから公開する挙動

## 11. アクセシビリティ

- アクションバーは明確な操作群として構成。ステータスバッジ・自動保存インジケータは視覚のみに頼らずテキストを持つ（インジケータは `role="status"`／`aria-live="polite"` 相当）。
- タイトルの見出し風入力は `<label>`（視覚的に隠してよい）を保持し、プレースホルダのみに依存しない。
- 公開パネルは Dialog として focus trap／Esc 閉じ／`aria-label` を備える（radix Dialog 準拠）。
- ヒーローのぼかし背景は `aria-hidden`、前景画像のみ `alt`。
- コントラストは既存の `check:contrast` を維持（新規の生 hex を持ち込まない）。

## 12. テスト方針

- `EditorPage.test.tsx`: 既存の保存/公開/変換ガードのテストを壊さない。追加で
  - 公開パネルを開き、カテゴリ未選択で公開ボタンが無効なこと、選択で有効化されること
  - 公開済み記事編集時に主ボタンが「更新を公開」になること
  - 自動保存インジケータの状態表示（可能な範囲で）
- `ArticleDetailPage.test.tsx`: ヒーローが表示されること（比率変更はクラスの差し替えなので既存アサーションを壊さない範囲で）
- `ArticleCard` を参照するテスト（各一覧）: サムネ比率変更でセレクタが壊れないか確認、壊れたら弱めずに追随。
- `check:contrast` / `typecheck` / `build` を緑に保つ。

## 14. 決定事項の要約

- レイアウト: 記事詳細体裁のキャンバス、本文前にフォームを置かない
- sticky アクションバー: 戻る／ステータス／自動保存インジケータ／下書き保存／公開する（公開済みは「更新を公開」）
- タイトル: 見出し風入力
- ヒーロー: 未設定=1 行トリガー、設定=16:9 contain＋ぼかし背景（詳細・入力で統一）
- カテゴリ/タグ: 公開パネルに集約、カテゴリ必須ガード（サーバー既存＋クライアント先回り）
- サムネ: 4:3 cover（16:9 画像は左右トリミング可）
- 既存の保存・変換・アップロード・公開ロジックは無改変
- DB スキーマ変更なし
