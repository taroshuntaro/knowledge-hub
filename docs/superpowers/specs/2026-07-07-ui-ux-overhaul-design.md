## 1. 目的と背景

Phase 2b-1 でトークン駆動のデザイン基盤（紙のエディトリアル基調・ライト/ダーク・AA 検証）を入れたが、コントローラーによる全 11 画面の監査で、問題は**トークンの上の「ナビゲーション構造」と「情報設計」**に集中していると判明した。本刷新はビジュアルコンパニオンで方向性を確定済みで、以下を実現する:

1. **アプリシェルをサイドバー型に**（トップバーのナビ飽和・モバイル崩壊・ロゴ折返しを解消）
2. **フィードの情報設計を強化**（記事サムネイル・カテゴリ・タグ・著者・日付・反応数を持つ統一カード、ピックアップの明確化）
3. **記事ヒーロー画像**を新機能として追加（設定・一覧サムネイル・詳細表示。比率 16:9 で統一）
4. **ブランド主役色をインディゴへ**（暖色の紙・墨は維持、主役色のみ変更）
5. 各画面の一貫性（ボタン階層・破壊的操作の区別・検索フォームの統合・空状態）を底上げ

監査で挙げた P0（構造）〜 P2（一貫性）を包括する。作業量が大きいため**実装は複数サブフェーズ/計画に分割**する（§10）。

## 2. 確定済みの視覚方向（ビジュアルコンパニオンでユーザー承認）

| 項目 | 決定 |
|---|---|
| アプリシェル | 左サイドバー型（デスクトップ）／モバイルはハンバーガー → ドロワー |
| サイドバー内容 | 閲覧（フィード/検索/ブックマーク）→ カテゴリ一覧（2 階層・色ドット）→ 作成（記事を書く/マイ記事）→ 管理（カテゴリ管理/ユーザー、admin のみ）→ 下部にアカウント（アバター＋名前＋設定/ログアウトのメニュー） |
| フィードカード | 統一カード＝**左端に横長 16:9 サムネイル**（カード高さいっぱい・下部余白なし）＋本文（カテゴリチップ・タイトル・抜粋・著者アバター/名前・日付・タグ・反応/コメント数） |
| サムネイル比 | **16:9 固定**。記事に設定するヒーロー画像の比＝一覧サムネイルの比を一致（トリミングで比率を変えない）。画像なしは**カテゴリ色のフォールバックタイル**（頭文字） |
| ピックアップ | 新着と**同じカード形**＋アクセント枠＋📌ラベルで、新着の上に縦並び。0 件は非表示・1 件はカード 1 枚に自然縮退 |
| ブランド主役色 | **インディゴ**（H≈258）。地・墨（暖色ニュートラル）は維持し、primary/accent/ring/カテゴリドット等の主役色のみ変更。ライト/ダーク両方を用意 |

## 3. デザイントークンの変更

`apps/web/src/index.css` のトークンのみ変更する（コンポーネントに生色は書かない原則は維持）。

- **維持**: `--background` `--foreground` `--card` `--muted-foreground` などの暖色ニュートラル（紙・墨）。フォント（システムサンセリフ、CDN 禁止）。radius。
- **変更（ライト）**: `--primary`（インディゴ oklch(0.5 0.14 258) 目安・白文字）／`--accent`（淡インディゴ地）／`--accent-foreground`（濃インディゴ）／`--ring`（インディゴ）。リンク色は primary 連動。
- **変更（ダーク）**: 墨チャコール地に**明るめのインディゴ**（ring/accent-foreground/link が沈まない明度）。primary は暗地に対して白/淡文字で AA。
- **カテゴリ色**: カテゴリごとの識別色（ドット/フォールバックタイル）は、決められた少数のパレット（トークン）から `category.id` で決定論的に割り当てる。生 hex は使わない。
- **不変条件**: 変更後、**全 fg/bg ペアを `scripts/check-contrast.mjs` で WCAG AA (4.5:1) 再検証**し、ライト/ダーク両方 green を維持する（既存の品質ゲート）。この検証にパスしない色は採用しない。

## 4. アプリシェル（レイアウト）

`apps/web/src/components/Layout.tsx` を全面刷新し、`Sidebar` コンポーネントを新設する。

- **デスクトップ（≥ md）**: 固定幅の左サイドバー（~220px）＋本文カラム。サイドバーはスクロール追従（sticky）。
- **モバイル（< md）**: サイドバーは非表示。上部に細いバー（☰ ＋ ワードマーク ＋「＋記事を書く」）。☰ で**ドロワー**（同一内容 ＋ scrim）を開閉。ドロワーはルート遷移で自動的に閉じる。
- **サイドバー構成**: §2 の順序。
  - **カテゴリ一覧**は `GET /api/categories`（既存 `listCategoryTree`）から取得し 2 階層で表示。各項目クリックで `/categories/:id`。末尾に「すべてのカテゴリ →」。
  - **アカウント**は下部に集約: アバター＋表示名。クリックで「設定 / ログアウト」メニュー（現状ヘッダーに散っている 設定・ログアウトをここへ）。
  - `管理` グループは admin ロールのみ表示（既存の権限判定を流用）。
- **通知ベル**は上部バー（モバイル）とサイドバー上部（デスクトップ）に配置。未読バッジは現状の仕組みを流用。
- **アクセシビリティ**: ドロワーの開閉ボタンに `aria-expanded`/`aria-controls`、フォーカストラップ、Esc で閉じる。ナビは `nav`＋`aria-current`。

## 5. フィードと統一記事カード

`ArticleCard` を作り直し、フィード（`HomePage`）・カテゴリ・タグ・著者・マイ記事・ブックマーク・検索結果の一覧すべてで共有する。

- **カード構造**: `<article>` 内に flex 行。左に 16:9 サムネイル（`~160px` 幅・カード高さいっぱい・`object-fit: cover`）、右に本文。
- **本文の要素**: カテゴリチップ（色）／タイトル（2 行まで）／抜粋（1〜2 行 clamp）／メタ行（著者アバター＋名前・相対でなく絶対日付・タグ 2〜3 個・♥ 反応数・💬 コメント数）。
- **サムネイルのフォールバック**: `heroImage` 未設定時はカテゴリ色地＋タイトル/カテゴリ頭文字のタイル。同じ 16:9 枠でレイアウトは不変。
- **ピックアップ**: `HomePage` 上部に「📌 ピックアップ」セクション（`listPickup` の結果）を同じカードにアクセント枠＋📌ラベルで縦並び。件数 0/1 の縮退は §2 の通り。
- **空状態**: 一覧が空のとき、柔らかい文言＋アイコン＋主要アクション（例: フィードが空 → 「最初の記事を書く」）。

### 5.1 一覧 API のデータ拡張（バックエンド）

現状の一覧項目（id/title/excerpt/authorId/authorName/categoryId/pinnedAt/publishedAt/updatedAt）に、カード表示に必要な項目を追加する。**カーソルページングの既存パターン・可視性ルールは変更しない**。

- 追加項目: `heroImage`（upload id or null）／`categoryName`（join）／`authorAvatarUrl`／`tags`（名前配列）／`reactionCount`／`commentCount`。
- **N+1 回避**: タグ・反応数・コメント数はページ内 `articleId` 集合に対する 1 クエリの集約（`in (...)` ＋ `group by`）でまとめて解決し、行にマージする。件数は概数でなく実数。
- 型は `packages/shared` に定義し、hono RPC で web が直接消費（既存方針）。

## 6. 記事ヒーロー画像（新機能）

**これはビジュアル変更でなく機能追加**であり、バックエンド・エディタ・表示の 3 面に及ぶ。既存の S3 互換アップロード基盤（magic-byte 検証・MIME 許可リスト・サイズ上限・nosniff）をそのまま流用する。

- **スキーマ**: `articles` に `hero_image_upload_id uuid null references uploads(id)` を追加（マイグレーション）。論理削除・リビジョン等の既存挙動には影響しない。
- **設定 UI（エディタ）**: タイトル付近に「ヒーロー画像」枠（**16:9 のプレースホルダ**）。画像を選択 → 既存アップロード API で保存 → `hero_image_upload_id` を設定。「削除」で null 化。**設定時の枠＝一覧サムネイルの見え方**（同じ 16:9・object-fit cover）。
- **配信**: 既存 `GET /api/uploads/:id`（認証必須・private キャッシュ）で配信。一覧は §5.1 の `heroImage` を、記事詳細は上部に 16:9 で表示。
- **可視性**: ヒーロー画像は記事本文と同じ可視性（下書き/削除は非公開経路のみ）。他人のアップロード UUID を設定できる既知 Minor と同じ性質だが機密影響はない（本文画像と同様）。
- **フォールバック**: 未設定はカテゴリ色タイル（§5）。

## 7. 主要画面の適用

すべて同じシェル・カード・トークン・ボタン階層に載せる。

- **記事詳細（`ArticleDetailPage`）**: 上部に 16:9 ヒーロー（未設定はフォールバック無しでタイトル直下から。＝詳細では大きな空タイルは出さず、あるときだけ表示）。タイトル下に**メタ情報**（カテゴリチップ・著者アバター＋名前・公開日・タグ）。「← フィードに戻る」導線。アクションは**ボタン階層を統一**（主: 編集／副: ブックマーク・ピン留め／破壊的: ゴミ箱へ は destructive スタイルのボタンに統一。現状の「赤いテキストリンク」をやめる）。反応・コメントは現状の機能を維持しつつ間隔・見出しを整える。
- **エディタ（`EditorPage`）**: §6 のヒーロー画像設定を追加。既存のタイトル/カテゴリ/タグ/モード切替/ツールバー/自動保存はそのまま、システムのスタイルへ。
- **検索（`SearchPage`）**: 現状**キーワード欄がグローバルヘッダーにあり、フィルタだけページにある**分断を解消。検索ページに**キーワード入力＋カテゴリ/タグフィルタを 1 つのフォーム**として置き、`q`/`category`/`tag` を URL クエリに同期（既知 Minor の URL 同期もここで解消）。サイドバーの検索はこのページへ遷移。結果は共有 `ArticleCard`。
- **ログイン/招待/リセット（`AuthShell`）**: ワードマークをブランドとして整え（プレーンテキスト → ロゴタイポ）、簡潔な製品タグライン、「または」区切りに罫線。SSO ボタン（Phase 4b）は現状維持。
- **設定（`SettingsPage`）**: サイドバーシェル下でカード群を整える。アバターのフォールバック（現在は赤い丸）を、他画面と同じイニシャル/カラーのアバターに統一。
- **ユーザー管理（`AdminUsersPage`）**: テーブルに行 hover・ゼブラ等の可読性、破壊的操作（無効化）を destructive スタイルへ。
- **カテゴリ管理 / マイ記事 / ブックマーク / 通知**: 同シェル・同カード・同ボタン階層へ。マイ記事のタブ、通知一覧は機能を維持。

## 8. 一貫性ルール（横断）

- **ボタン階層**: 主（primary 塗り）＝各画面の主アクション 1 つ／副（outline）＝補助／破壊的（destructive）＝削除・無効化。1 画面に primary 塗りは原則 1 つ。
- **破壊的操作**は必ず destructive スタイル＋（不可逆なものは）確認。「赤いテキストリンク」の混在を廃止。
- **アバター**は 1 コンポーネントに統一（画像 or イニシャル＋カテゴリ非依存の安定色）。設定・カード・詳細・管理で共通。
- **空状態**は「アイコン＋柔らかい一文＋主要アクション」の型に統一。
- **日付**は絶対表記（例: 2026年7月5日）に統一。

## 9. 制約・非スコープ

- **トークンが唯一の色の源泉**（生 hex 禁止）／システムフォントのみ（CDN 禁止）／ライト・ダーク両対応で全ペア AA。
- 既存の**機能・API 契約・可視性/権限ルール・カーソルページング**は原則変更しない。変更するのは §5.1（一覧項目の追加・後方互換）と §6（ヒーロー画像の追加）のみ。
- 既存の web テストは role/label ベース。刷新でセレクタや構造が変わる箇所はテストも更新する（E2E は本刷新後の Phase 4c で書くため、今は単体/統合の green 維持を受け入れ基準とする）。
- **非スコープ**: 記事本文の編集機能そのもの（Tiptap 挙動）／検索エンジン（pg_bigm）／通知ロジック／認証フロー。これらは見た目の適用のみ。RAG・カテゴリ ACL 等の将来機能は扱わない。

## 10. 実装分割（計画フェーズで詳細化）

大きいため、動作する単位で分割する（各サブフェーズが単体で回帰 green・マージ可能）。**本刷新はロードマップ上「UI/UX 刷新」フェーズとして E2E + 仕上げ（旧 Phase 4c）の前に置く**（正式なフェーズ番号は計画着手時に確定）:

- **S1: デザイントークン（インディゴ）＋アプリシェル** — トークン差し替え＋ AA 再検証、`Sidebar`＋モバイルドロワー、Layout 刷新。機能・データ変更なし。既存テスト（role/label）を更新して green。
- **S2: ヒーロー画像（バックエンド＋エディタ）** — スキーマ/マイグレーション、アップロード連携、一覧 API のデータ拡張（§5.1）、エディタの 16:9 設定 UI。実 DB テスト。
- **S3: 統一カード＋フィード/ピックアップ＋一覧各画面** — `ArticleCard` 共有、HomePage（ピックアップ/新着）、カテゴリ/タグ/著者/マイ記事/ブックマーク/検索結果への適用。
- **S4: 記事詳細＋エディタ＋検索統合＋認証/設定/管理＋一貫性パス** — 残り画面の適用とボタン階層/破壊的操作/空状態/アバター統一。
- 各サブフェーズ完了後 `pnpm run verify`（typecheck＋テスト＋contrast＋build＋audit）green。

（注: 本刷新の後に別途 E2E + 仕上げフェーズ（旧 Phase 4c）で Playwright 主要フローを書く。全体レビュー由来の M-6 SMTP 本番対応・M-7 本番設定必須化/HSTS はその仕上げ側で対応。）

## 11. 受け入れ基準

- 全 11 画面がサイドバーシェル・インディゴ・統一カード/ボタンで一貫。
- モバイルでナビが崩れずドロワーで操作可能。
- フィード/一覧カードにサムネイル（16:9・フォールバック）・カテゴリ・タグ・著者・日付・反応/コメント数が出る。
- ピックアップが新着と同じカードで区別表示され、0/1 件で縮退。
- 記事にヒーロー画像を設定でき、設定枠と一覧サムネイルの見え方が一致。
- 検索がキーワード＋フィルタの単一フォーム＋ URL 同期で動作。
- `check:contrast` がライト/ダーク両方 green、`pnpm run verify` exit 0。
