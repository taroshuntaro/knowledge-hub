# Phase 2b 設計: デザイン基盤刷新 + リッチエディタ UI

- 日付: 2026-07-05
- ステータス: 承認済み
- 関連: [knowledge-hub 全体設計](./2026-07-04-knowledge-hub-design.md)（エディタ挙動は同書 §6 が正）

## 目的と背景

Phase 2a までの UI は機能検証用の最小スタイル（手書き CSS 29 行）であり、製品としての統一感がない。Phase 2b ではリッチエディタ（Tiptap）導入と同時にデザイン基盤を刷新し、以後のすべての画面が同一のデザインシステムの上に載る状態を作る。

エディタの挙動仕様（Markdown 無損失往復・対応記法の限定・往復変換テスト）は全体設計 §6 を正とし、本書は**デザインシステムと UI 面**を定義する。

## 決定事項（ユーザー確定）

| 項目 | 決定 |
|---|---|
| UI 基盤 | shadcn/ui + Tailwind CSS v4（コンポーネントはコード所有） |
| トーン | エディトリアル基調（読み物中心）+ 親しみのアクセントを少量 |
| テーマ | ライト / ダーク両対応 |
| 適用範囲 | 全 8 画面を刷新 |
| 進め方 | 2b-1「デザイン基盤 + 全画面刷新」→ 2b-2「Tiptap リッチエディタ」の 2 ブランチ |

## 実装分割

- **2b-1: デザイン基盤 + 全画面刷新** — 機能変更ゼロの見た目刷新。API・ルーティング・状態管理は一切変更しない。既存 web テスト（role/label ベース）がそのまま回帰テストとして green を維持することが受け入れ基準。
- **2b-2: Tiptap リッチエディタ** — 完成したデザインシステムの上に、切替式エディタ（リッチ ⇔ Markdown ソース）と画像 D&D を実装。挙動は全体設計 §6 に従う。

実装計画は各ブランチ着手時に just-in-time で作成する（本書は両方をカバーする単一スペック）。

## 技術基盤（2b-1）

- **Tailwind CSS v4**: `@tailwindcss/vite` プラグイン、CSS-first 設定（`@theme`）。
- **shadcn/ui**: CSS 変数モード。コンポーネントは `apps/web/src/components/ui/` にコピー所有（外部ライブラリとしてのバージョン追従はしない）。
- `@/` パスエイリアスを tsconfig + Vite resolve に追加（shadcn の import 規約）。
- アイコンは lucide-react（npm バンドル、外部通信なし）。
- 既存 `styles.css` は全廃し、`index.css`（トークン定義 + Tailwind）に置き換える。移行中の新旧 CSS 並存は 2b-1 ブランチ内でのみ許容し、マージ時点で旧 CSS は残さない。

### デザイントークン

- **トークンが唯一の色の源泉**。shadcn 標準のセマンティックトークン（`--background` `--foreground` `--primary` `--muted` `--accent` `--destructive` `--border` `--ring`、radius）を採用し、生の色コードをコンポーネントに直書きしない。
- 記事本文用に prose（タイポグラフィ）設定をトークン連動でカスタマイズ（`@tailwindcss/typography` を使用）。
- CodeMirror（2b-1）および Tiptap（2b-2）のエディタテーマもトークンに連動させ、ライト/ダーク切替に追従させる。

### ダークモード

- `.dark` クラス方式（`<html>` に付与）。
- ヘッダーにトグルを設置。初期値は `prefers-color-scheme`、ユーザー選択は localStorage に永続化。
- **FOUC 防止**: テーマ初期化は `index.html` のインラインスクリプトで React マウント前に行う。

## ビジュアルの方向性と制約

- **エディトリアル基調**: 本文の読みやすさ最優先。本文幅 ~42rem、`line-height` 1.8 前後、明確な見出し階層。
- **フォントはシステムスタックのみ**（Hiragino Sans / Yu Gothic / Noto Sans JP フォールバックのサンセリフ）。オンプレ（閉域）前提のため**外部 CDN からの Web フォント読み込みは禁止**。将来 Web フォントを使う場合もセルフホストとする。
- **親しみのアクセントは限定的に**: 温かみのある 1 アクセントカラー、タグはピル形状、カードの角丸、空状態は柔らかい文言 + アイコン。装飾過多にしない。
- **コントラストはライト/ダーク両テーマで WCAG AA** を満たす。
- 具体的な配色・タイポの確定は 2b-1 の最初のタスク（デザイン基盤タスク）で frontend-design スキルを用いて行う。**このタスクの成果物 = トークン定義一式 + フィード画面への適用サンプル（ライト/ダーク両テーマ、AA コントラスト検証済み）**。承認後に他画面へ展開する。

## 画面刷新の範囲（2b-1）

全 8 画面: ログイン / 招待受諾 / パスワードリセット、フィード（ピックアップ + 一覧）、記事詳細、マイ記事、エディタ（現行 CodeMirror ソースエディタの見た目のみ刷新）、カテゴリ管理、ユーザー管理、設定。

- 管理系画面（カテゴリ・ユーザー管理）は shadcn 標準コンポーネントの素組みとし、デザイン工数を抑える。
- レイアウトはヘッダー + コンテナ構成を維持しつつ刷新。
- 375px 幅でも崩れないこと（モバイル専用最適化までは踏み込まない）。
- ローディング（skeleton）・空状態・エラー表示もトークン準拠で統一する。

## エディタ UI（2b-2）

- ツールバーは shadcn コンポーネント（Toggle / ToggleGroup / Tooltip / DropdownMenu / Separator）で構成。
- リッチ ⇔ Markdown ソースの切替タブ、画像 D&D のドロップゾーンもトークン準拠。
- 挙動（往復無損失・対応記法・`/api/uploads` 連携）は全体設計 §6 に従い、本書では規定しない。

## 検証・受け入れ基準

### 2b-1

1. 既存 web テストが無修正（またはセレクタの機械的追従のみ）で green。
2. root typecheck + web 本番ビルドがクリーン。
3. Playwright スクリーンショットでライト/ダーク両テーマの全画面目視確認パスを実施。
4. トークンの AA コントラスト検証（デザイン基盤タスクの完了条件）。

### 2b-2

全体設計 §6 の往復変換テストスイート + エディタ UI テスト（詳細は 2b-2 実装計画で定義）。

## 非スコープ

- ロゴ・ブランディング制作（ワードマークの文字スタイリングは可）
- メールテンプレートのデザイン
- モバイル専用最適化
- 検索・コメント・リアクション等の Phase 3 機能
